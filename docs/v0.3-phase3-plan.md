# IKKI v0.3 フェーズ3 実装計画（企業ダッシュボード・B2B）

> フェーズ1（バイラル診断）・2（評価の深化）は本番稼働済み。本書はフェーズ3。
> 合意後に着手し、各コミットで build/lint を通して分割・デプロイ可能に保つ。

## 0. 遵守する前提（v0.3ガードレール）
1. フェーズ順守（フェーズ3の中も F3-1認証基盤 → F3-2/F3-3 → F3-4 の順）
2. **求人情報の無差別クローリングをしない**。企業の構造化入力＋許可されたソースのみ（robots.txt/規約尊重）
3. 全スコアに根拠発言を紐付ける（v0.2原則の維持）。推薦/非推薦は根拠つき
4. 氏名・連絡先・ログ全文は「マッチ成立 かつ 本人同意」時のみ企業に開示（RLSで担保）

## 1. 現状で使える資産
- `career_cards`（profile, quality.completeness, transcript, log_disclosure_consent）… 候補者評価の母体
- `lib/grounding.ts` / 根拠ドリルダウン（`components/card.tsx`）… 推薦理由の根拠表示に流用
- `lib/competencyModel.ts`（5軸×BARS）… 企業の合格基準の下限点を同じ軸で設定
- `lib/deviation.ts`（偏差値・参考値閾値）… オファー条件の相対評価(F3-4)に再利用
- Supabase（本番設定済み・service_role サーバーアクセス）… ここに **Auth + RLS** を本格導入

## 2. 現状のギャップ（フェーズ3で新規に必要）
- **認証基盤が無い**（現状は匿名 + 端末localStorage）。企業ユーザーを識別・分離する仕組み
- **RLSポリシーが無い**（現状RLS有効・ポリシー無し＝service_role専用）。候補者/企業で見える範囲を分ける
- 企業ロール・企業アカウント・所属のデータモデル
- 合格基準テンプレート、候補者の基準照合エンジン、推薦/条件付/非推薦の分類
- オファーの構造化入力、オファー条件の相対評価

## 3. フェーズ3 スコープと実装順序

### F3-1 企業アカウントと認証（最初・基盤）
- **Supabase Auth を導入**（メール+パスワード / マジックリンク）。企業ユーザーのロールを追加し求職者と分離
- **ここで初めて認証を本格化する**（フェーズ1/2は匿名+端末保存のまま維持）
- 追加スキーマ:
```sql
create table if not exists companies (
  id uuid primary key default gen_random_uuid(),
  name text not null,
  created_at timestamptz default now()
);
create table if not exists company_members (
  user_id uuid primary key,          -- auth.users.id
  company_id uuid references companies(id),
  role text not null default 'member',
  created_at timestamptz default now()
);
```
- **RLS方針**:
  - `career_cards`：企業は**匿名要約（氏名・transcript除外）**のみ閲覧可。氏名・ログ全文は「マッチ成立 かつ log_disclosure_consent」時のみ。ビュー or ポリシーで列を絞る
  - 候補者の端末フロー（匿名）は現状維持（service_role経由のAPIはそのまま）
- 企業ログイン後の土台となる `/company` ルート（認証ガード）

### F3-2 合格基準テンプレ + 候補者の自動分類
- 企業が**合格基準を登録**：必須コンピテンシーの下限点（5軸）/ 必須書類 / 希望特性
- 候補者カードを基準に照らし、**推薦 / 条件付き / 非推薦**を**根拠つき**で自動提示（v0.2ドリルダウンで発言原文へ）
- 「基準を満たす候補者◯名」「一次面接スキップで削減できる想定工数」を可視化
- スコア順一覧・一括アクション（ポテンシャル大量採用向け）
- 追加スキーマ:
```sql
create table if not exists hiring_criteria (
  id uuid primary key default gen_random_uuid(),
  company_id uuid references companies(id),
  name text not null,
  min_competencies jsonb not null default '{}',  -- {problem_solving:3,...}
  required_documents jsonb not null default '[]', -- ['resume']
  preferred_traits jsonb not null default '[]',
  created_at timestamptz default now()
);
```

### F3-3 オファーと条件の構造化入力（クローリングの代替）
- 企業はオファー時に **給与・福利厚生・仕事内容を構造化フォームで入力**（自由記述でなく項目化）。これが条件データの第一ソース
- 任意で、企業が自社の公開採用ページURLを登録した場合のみ、**その1ページだけ**取得して下書き補助（robots.txt/規約尊重、無差別クロール禁止）
- 追加スキーマ（仕様どおり）:
```sql
create table if not exists offers (
  id uuid primary key default gen_random_uuid(),
  company_id uuid references companies(id),
  card_id uuid references career_cards(id),
  salary_min int, salary_max int,
  benefits jsonb,
  role_description text,
  status text default 'sent',
  created_at timestamptz default now()
);
```

### F3-4 オファー条件の相対評価（求職者を惹きつける）
- 蓄積された offers を母集団に、給与・福利厚生の**上位何%（好条件度）**を算出（`lib/deviation.ts` 流用）
- 求職者のオファー受信画面で条件をグラフィカルに即表示 +「この条件は全体の上位◯%」バッジ
- サンプル不足時は F1-4 同様「参考値」表示

## 4. 判断ポイント（合意済み・2026-07-22）
- 4-1 認証範囲: **企業側のみ認証**。求職者は匿名+端末保存を維持 → ✅
- 4-2 候補者プール: **本人が「企業に公開」同意したカードのみ**（`career_cards.discoverable`）→ ✅
- 4-3 公開ページ取得: **今回は構造化入力のみ**（URL取得は将来）→ ✅
- 4-4 順序: **F3-1 → F3-2 → F3-3 → F3-4** → ✅

## 4'. 判断ポイント（詳細・参考）

### 4-1. 認証の範囲（最重要）
- **A案（推奨）**: Supabase Auth を**企業側のみ**に導入。求職者は現状の匿名+端末保存を維持（フェーズ3は「企業がカードを見る」ためのB2B。求職者ログインは今回作らない）
- B案: 求職者もログイン化（本人が別端末でカードを見られる等）。範囲が大きく増える

### 4-2. 候補者カードの企業への「見せ方」
- 現状カードは端末所有（匿名UUID）。企業ダッシュボードで**どの母集団を見せるか**:
  - **A案（推奨・現実的）**: 候補者が「企業に公開する」と明示同意したカードのみを企業プールに載せる（`career_cards` に `discoverable boolean` を追加）。同意ベースで健全
  - B案: 全カードを匿名で企業に見せる（同意設計が重くなる）

### 4-3. 公開採用ページ取得（F3-3の補助機能）
- **A案（推奨）**: フェーズ3では**構造化入力のみ**実装し、URL取得は将来に回す（robots.txt尊重の実装コストと法的配慮を後回しにできる）
- B案: 今回、許可ページ1枚の取得も実装する

### 4-4. 実装順序
- **F3-1（認証+RLS基盤）→ F3-2（基準+分類）→ F3-3（オファー構造化入力）→ F3-4（相対評価）** で良いか

## 5. コミット分割案（各 build/lint 通過・デプロイ可能）
1. F3-1a: companies/company_members スキーマ + Supabase Auth 導入 + `/company` 認証ガード
2. F3-1b: RLS ポリシー + 企業向け「匿名カードプール」取得API（同意カードのみ・氏名/ログ除外）
3. F3-2a: hiring_criteria 登録UI + 基準照合エンジン（推薦/条件付/非推薦・根拠つき）
4. F3-2b: 候補者一覧（スコア順・基準充足）+ 削減工数の可視化 + 一括アクション
5. F3-3: offers 構造化入力フォーム + 送信
6. F3-4: オファー条件の相対評価（好条件度・参考値バッジ）+ 候補者側の受信表示
7. スキーマ/README/デプロイ確認

## 6. 受け入れ基準（フェーズ3）
- 企業が合格基準を登録すると、候補者が推薦/条件付/非推薦に**根拠つき**で分類される
- 氏名・ログ全文が**本人同意なしに企業へ出ない**（RLSで担保）
- オファー条件が構造化入力され、相対評価バッジが出る（サンプル不足時は参考値）

## 7. リスク・留意
- 認証+RLSは**セキュリティ的に最も重い**。RLSポリシーの取り違えは情報漏洩に直結するため、公開範囲の列を絞ったビュー（匿名プール）を用意し、氏名/transcriptは別ゲートにする
- フェーズ3はデータ（母集団）が育ってから価値が出る。**機能は用意しつつ、初期は空でも壊れない**設計にする（偏差値の参考値フォールバックと同じ思想）
- 企業/求職者の本人確認・不正利用対策は本計画の範囲外（別途必要）
