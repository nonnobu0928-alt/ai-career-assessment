## 0. 遵守する前提（ガードレール）

1. **フェーズ順守** — 1→2→3。跨いだ先行実装をしない。
2. **動画/音声** — 表情・声質から合否/性格を推定しない。発話内容評価（根拠引用つき）＋伝わりやすさの参考指標のみ。
3. **求人情報** — 無差別クローリングをしない。企業の構造化入力＋許可されたソースのみ。
4. **根拠主義（v0.2原則）** — 全スコアに根拠発言を紐付ける。ハルシネーション照合・誠実な欠損を維持。

## 1. 現状分析（何が既にあるか）

### 技術スタック
- Next.js 16.2 (App Router) / React 19 / TypeScript / Tailwind v4 / recharts
- `@anthropic-ai/sdk`（面接・解析、structured outputs + サーバー照合）
- `@supabase/supabase-js`（サーバー専用 service_role、DB任意）

### ルーティング / UI
- **単一ルート `/`**。`app/page.tsx` が巨大クライアントコンポーネントで、画面を state で切替（landing / form / interview / analyzing / result）。
- **動的ルート・公開ページ・OGP は未実装**。カード表示は `components/card.tsx`。

### データモデル（`supabase/schema.sql`）
- `career_cards`（id, created_at, name, role, years, transcript, profile(jsonb), log_disclosure_consent, quality(jsonb)）
- `interview_sessions`（id, card_id, state(jsonb), created_at）
- RLS は有効だがポリシー無し＝**service_role 経由のサーバーアクセスのみ**。認証なし。
- 端末は `localStorage` にカードUUID（または本体）を保持。**カード取得はUUIDを知る本人だけ**というデモ用モデル。

### 再利用できる資産（v0.3で活きるもの）
- `lib/grounding.ts`（引用逐語照合・数字の発言由来照合）→ **コミュ試験・動画評価にそのまま流用**
- `lib/competencyModel.ts`（5軸×BARS）→ コミュ試験/動画の評価軸の型を踏襲
- `lib/quality.ts`（充足度スコア）→ フェーズ2で書類/試験/面接フラグを合算して拡張
- `lib/theme.ts`（藍×朱印トークン）→ OGP・診断UIのブランド踏襲
- `lib/interviewEngine.ts` / `questionBank.ts` → コンピテンシー層（既存）はそのまま「本体診断」に使う

### 現状のギャップ（v0.3で新規に必要）
- 複数ルート／動的セグメント（公開ページ・OGP）
- 匿名で拡散するための **公開共有ID** と **公開専用の匿名要約API**
- 診断の**中断・再開**（quiz層のセッション永続化）
- **匿名集計・偏差値**の基盤
- 演出（framer-motion）と reduced-motion フォールバック
- OGP画像の動的生成

## 2. フェーズ1 実装計画（最優先・詳細）

### 2-0. スコープの明確化（重要）
仕様 F1-1 は3レイヤー構成を掲げるが、フェーズ分割（§0-1）では **MBTI風・特性層はフェーズ2（F2-2）**。跨ぎ禁止のため、**フェーズ1では特性層を実装しない**。フェーズ1の対象は次の4つ：
- **クイック層**（F1-1 レイヤー1：選択式中心 3〜4分）
- **ゲーミフィケーションUI**（F1-2）
- **結果画面 + シェア/OGP + 公開ページ**（F1-3）
- **偏差値・相対評価エンジン**（F1-4）
- **コミュニケーション能力テスト**（F1-5）

※既存の v0.2 コンピテンシー面接（レイヤー2）は「続けて深掘りする本体」として導線を繋ぐのみ（再実装しない）。

### 2-1. 不足技術要素
| # | 要素 | 内容 |
|---|---|---|
| 1 | ルーティング再編 | `/`(LP)、`/quiz`(診断フロー)、`/result/[cardId]`(本人結果)、`/r/[shareId]`(公開ページ)、`/result/[cardId]/opengraph-image.tsx`(OGP) を App Router で新設。既存の巨大 `page.tsx` は段階的に分割（まずLP/面接は現状維持、診断だけ新ルート） |
| 2 | 公開共有ID | 本人カードUUIDとは**別トークン** `share_id`(nanoid) を発行。公開ページのキーにUUIDを使わない（本人取得キーの漏洩防止） |
| 3 | 診断セッション永続化 | クイック層の途中回答を保存し中断・再開。`interview_sessions` は面接専用のため、汎用の `diagnostic_sessions` を新設（DB無し時は localStorage フォールバック） |
| 4 | 匿名集計エンジン | `score_distributions` に確定スコアを匿名で追加し、偏差値/パーセンタイルを算出。個人特定情報は持たない |
| 5 | OGP動的生成 | `@vercel/og`(satori) でスコア/タイプ/偏差値/キャッチを載せた画像を生成。**日本語フォントの埋め込みが必要**（satoriはブラウザに頼らずフォントバイナリを渡す）。※後述の判断ポイント参照 |
| 6 | 演出基盤 | `framer-motion` 導入。1問1画面遷移・スコア加算・カウントアップ。`prefers-reduced-motion` で即時表示にフォールバック |
| 7 | クイック層設問＆採点 | 選択式の**正誤・加点根拠をコード側に固定**（`lib/quizBank.ts` 新設）。恣意性排除 |
| 8 | コミュ試験評価 | シチュエーション別に評価軸＋BARSを新設（`lib/commTest.ts`）。自由記述は Anthropic で採点、**根拠となる本人の記述を必ず引用**（grounding流用） |

### 2-2. 追加スキーマ（フェーズ1）
```sql
-- 匿名スコア分布（仕様F1-4のとおり）
create table if not exists score_distributions (
  metric text primary key,             -- 'competency_overall' | 'communication' | 'quick_overall' 等
  samples jsonb not null default '[]',  -- 匿名スコア配列（個人特定情報なし）
  updated_at timestamptz default now()
);

-- 汎用診断セッション（クイック層の中断・再開）
create table if not exists diagnostic_sessions (
  id uuid primary key default gen_random_uuid(),
  kind text not null,                  -- 'quick' | 'comm_test'
  state jsonb not null default '{}',
  card_id uuid references career_cards(id),
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);
alter table diagnostic_sessions enable row level security;

-- career_cards への追加列
alter table career_cards add column if not exists share_id text unique;         -- 公開ページのキー（nanoid）
alter table career_cards add column if not exists quick_scores jsonb;           -- クイック層スコア
alter table career_cards add column if not exists communication jsonb;          -- コミュ試験結果（軸別スコア＋根拠引用）
alter table career_cards add column if not exists deviation jsonb;              -- 算出時点の偏差値/パーセンタイル（metric別）
```
> `deviation` は「表示時点のスナップショット」。母集団は更新され続けるため、確定値ではなく再計算前提。公開ページは最新の `score_distributions` から都度算出してもよい（実装時に選択）。

### 2-3. 推奨ライブラリ / API（フェーズ1）
| 用途 | 選定 | 理由 |
|---|---|---|
| 演出 | `framer-motion` | 仕様指定。reduced-motion対応が容易 |
| OGP画像 | `@vercel/og`（内部 satori/resvg） | 仕様指定。Next.js公式・エッジ生成 |
| 共有ID | `nanoid` | URL安全・短い・衝突しにくい。UUIDと分離 |
| 偏差値計算 | 自前（ライブラリ不要） | 平均・標準偏差・パーセンタイルのみ |
| コミュ試験採点 | 既存 `@anthropic-ai/sdk` | v0.2の根拠引用・照合をそのまま流用 |

### 2-4. 受け入れ基準への対応
- 「診断を最後まで→シェア画像生成→公開ページ」→ F1-3（OGP + `/r/[shareId]`）
- 「公開ページに発言ログ・企業評価が出ない」→ 公開APIは**匿名要約のみ**返す（タイプ名・偏差値バッジ・強み3つ）。`transcript`/企業向け項目は返却しない
- 「サンプル不足時は参考値」→ F1-4（`samples.length < 100` で参考値バッジ、確定表示しない）
- 「reduced-motion で演出無効」→ F1-2

### 2-5. コミット分割案（各コミットで build/lint 通過・デプロイ可能）
1. **基盤**: `framer-motion`/`nanoid`/`@vercel/og` 追加、ルーティング土台、reduced-motion フック、診断UIの共通部品

5. **結果＆シェア**: 結果画面、`/result/[cardId]/opengraph-image.tsx`、公開ページ `/r/[shareId]`、「友達も診断」CTA

## 3. フェーズ2 概要（先行実装しない・洗い出しのみ）
- **F2-1 履歴書パース**: `documents` テーブル。PDF抽出（`unpdf` or `pdf-parse`）＋画像OCR（Anthropic vision を第一候補、重い場合 `tesseract.js`）→ Anthropicで構造化 → **本人確認・編集で確定**（AI値と確定値をDBで区別）。事実裏取り用途に限定。
- **F2-2 MBTI風特性**: オリジナル設問（既存MBTIの商標・設問を流用しない）4〜6軸。**合否非使用**、企業ビューでは「参考特性」に隔離。
- **F2-3 動画/音声面接**: 音声MVPから。録画前に用途・保存範囲の同意取得。文字起こしAPI選定（候補：OpenAI Whisper系 / Deepgram / AssemblyAI。日本語精度・コスト・保持ポリシーで比較、選定理由をREADMEに記載）。内容評価は既存コンピテンシー（根拠引用）、補助指標（話速・間・フィラー率）は参考値。**表情/声質推定はしない**。
- **F2-4 品質スコア拡張**: `lib/quality.ts` に書類提出済/コミュ受験済/面接受験済フラグを合算し「一次代替充足度」を1指標化。

## 4. フェーズ3 概要（先行実装しない・洗い出しのみ）
- **F3-1 企業アカウント + 認証**: Supabase Auth + RLS 本格導入、求職者/企業ロール分離。氏名・連絡先・ログ全文は「マッチ成立 かつ 本人同意」時のみ開示。
- **F3-2 合格基準テンプレ + 自動分類**: 企業が下限点・必須書類・希望特性を登録 → 推薦/条件付/非推薦を**根拠つき**で提示（v0.2ドリルダウンで発言原文へ）。
- **F3-3 offers 構造化入力**: `offers` テーブル。給与・福利厚生・職務を項目化入力。公開採用ページは企業登録時のみ1ページ取得（robots.txt/規約尊重、無差別クロール禁止）。
- **F3-4 オファー条件の相対評価**: offers 母集団から好条件度（上位何%）を算出、サンプル不足時は参考値。

## 5. 判断ポイント（合意済み・2026-07-21）
1. **公開共有ID**: 本人カードUUIDと分離し `share_id`(nanoid) を新設。→ ✅ 分離で確定。
2. **OGP の縦書き表現**: 公開画像は**横組みベースのレイアウト**とし、ブランドは色＋朱印モチーフで踏襲。→ ✅ 横組み+朱印で確定（satori縦書き非対応リスク回避）。
3. **DB前提**: **DB未設定時はシェア機能を無効化**し、診断はローカル完結（結果は見られるが公開URLは出さない）。→ ✅ 未設定時シェア無効で確定。
4. **偏差値の参考値閾値**: サンプル < 100 を「参考値」。→ ✅ 100で確定。
5. **設問内容の作成**: クイック層／コミュ試験の設問・BARSは当方でドラフト作成 → レビュー。→ ✅ ドラフト→レビューで確定。
6. **フェーズ1で特性層(MBTI風)を作らない**（フェーズ2へ）。→ ✅ フェーズ2送りで確定。
