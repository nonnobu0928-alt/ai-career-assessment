# IKKI v0.3 フェーズ2 実装計画（評価の深化）

> フェーズ1（バイラル診断）は本番稼働済み。本書はフェーズ2の計画。
> 合意後に着手し、各コミットで build/lint を通して分割・デプロイ可能に保つ。

## 0. 遵守する前提（v0.3ガードレール）
1. フェーズ順守（2の中も F2-1→F2-2→F2-4→**F2-3は最後**）。跨いだ先行実装をしない
2. 動画/音声は**表情・声質から合否/性格を推定しない**。発話内容評価（根拠引用つき）＋伝わりやすさの参考指標に限定
3. 求人情報の無差別クローリングはしない（フェーズ3）
4. 全スコアに根拠発言を紐付ける（v0.2原則の維持）

## 1. 現状で使える資産
- `lib/grounding.ts` … 音声面接・書類の主観評価回避に流用（発話評価は既存コンピテンシー照合）
- `lib/competencyModel.ts` … 音声面接の内容評価軸をそのまま使う
- `lib/quality.ts` … F2-4で提出フラグを合算して拡張
- `lib/commTest.ts` / `quizBank.ts` … 固定軸＋コード側採点の実装パターンをMBTI風に踏襲
- `app/api/*` … structured outputs + サーバー照合の実装パターン
- Supabase（本番設定済み）… `documents` 等の新テーブルを追加

## 2. フェーズ2 スコープと実装順序

### F2-1 履歴書・職務経歴書のパース（最初）
- **目的**: 「基礎情報の充足証明」。事実項目（氏名/学歴/職歴/在籍期間/資格）の裏取りに限定。**経歴の主観評価には使わない**
- **フロー**: アップロード(PDF/画像) → テキスト抽出・構造化 → **本人が確認・編集して確定** → AI値と本人確定値をDB上で区別
- **不足技術要素**: ファイルアップロードUI、抽出・構造化API、確認編集UI、提出済みフラグ
- **追加スキーマ**（仕様どおり）:
```sql
create table if not exists documents (
  id uuid primary key default gen_random_uuid(),
  card_id uuid references career_cards(id),
  kind text not null,               -- 'resume' | 'cv'
  parsed jsonb,                     -- 構造化結果(AI抽出)
  confirmed jsonb,                  -- 本人が確定した値(AI値と区別)
  confirmed_by_user boolean default false,
  created_at timestamptz default now()
);
```

### F2-2 MBTI風 特性診断（次）
- 独自設問（既存MBTIの商標・設問は流用しない）で **4軸** のタイプを判定。**コード側で固定採点**（quiz/commと同じ思想）
- **合否には使わない**。「チームでの動き方」「向いていそうな役割」の参考情報
- 企業ビューでは「参考特性」セクションに隔離し、評価スコアと混同させない。**偏差値は出さない**（F1-4の原則）
- 追加: `lib/traitModel.ts`（軸・設問・タイプ）、`/traits` ルート、結果は `career_cards.traits`（or diagnostic保存）

### F2-4 カード品質スコア拡張（次）
- `lib/quality.ts` に **書類提出済 / コミュ試験受験済 / 面接受験済** の充足フラグを合算
- 「このカードは一次面接を代替できる充足度か」を1指標に集約（企業の足切り・並べ替え用の土台）
- 既存 QualityScore に `completeness` 系フィールドを追加（後方互換を保つ）

### F2-3 動画/音声 AI面接（**フェーズ2の最後**・音声MVP）
- **取得**: 短い設問への回答を録音。**録画前に用途・保存範囲を明示して同意取得**
- **解析（主軸）**: 文字起こし → 発話内容を既存コンピテンシー基準で評価（根拠は発話の逐語引用・grounding流用）
- **補助指標（参考値・合否に直結させない）**: 話速・間・フィラー率など「伝わりやすさ」。スコアではなくフィードバックとして本人に返す
- **やらないこと**: 表情・容姿・声質からの性格/合否推定（実装しない）
- 企業ビューは「発話内容の評価（根拠つき）」を主に見せ、補助指標は明確に区別
- 追加: 録音UI（MediaRecorder）＋同意画面、文字起こし、`career_cards.voice` 保存

## 3'. 判断ポイント（詳細・参考）

### 3-1. 履歴書パースの方式
- **A案（推奨）**: Anthropic の文書/画像読み取り（PDF・画像をClaudeに渡してstructured outputsで構造化）。**新規依存・新規APIキー不要**、既存SDKで完結
- B案: 専用ライブラリ（PDF抽出 `unpdf` ＋ 画像OCR）を追加

### 3-2. 音声の文字起こし手段（コスト/キーに直結）
- **A案（推奨）**: ブラウザ標準 `SpeechRecognition`（音声入力で既に使用中）。**新規APIキー・課金不要**。話速/フィラー率は録音時間＋文字起こしから算出
- B案: 外部文字起こしAPI（OpenAI Whisper / Deepgram 等）。日本語精度は上がるが**別途APIキーと課金が必要**（直近のキー/課金トラブルを踏まえ、まずは無料のA案でMVPを推奨。将来差し替え可能な形にする）

### 3-3. MBTI風の軸数と設計
- **推奨**: 4軸・オリジナル設問。当方がドラフト作成 → レビュー（フェーズ1と同じ進め方）

### 3-4. 実装順序
- 仕様どおり **F2-1 → F2-2 → F2-4 → F2-3** でよいか（動画/音声は最後）

## 5. 受け入れ基準（フェーズ2・自己確認）
- 履歴書の抽出結果は**本人が確認・編集して確定**でき、AI値と確定値が区別される
- MBTI風は合否に使われず、企業ビューで「参考特性」に隔離される
- 音声面接は録画前に同意を取り、内容評価は根拠引用つき、補助指標は「参考」と明示され合否に直結しない
- 品質スコアに提出/受験の充足が反映される
